## 1. はじめに

### 1.1 概要

本仕様書は「贈与経済 2.0」[^1]のコンセプトに基づく**感謝経済プロトコル**を定義する。
感謝経済プロトコルは、人と人の間に生じる感謝の行為をデジタルに記録・流通させ、
それを経済的・社会的価値として可視化するためのオープンプロトコルである。

特定の実装・企業に依存せず、W3C DID / VC 等の国際標準に準拠することで、
複数の組織・システムが相互運用可能な形で接続できることを設計原則とする。

### 1.2 設計原則

| 原則 | 内容 |
|------|------|
| **公開性** | 感謝の社会的事実は原則として公開される。これが信頼の基盤となる |
| **本人コントロール** | 感謝の具体的内容は当事者が開示範囲を管理する |
| **プライバシー保護** | メッセージ内容は送信者が所有し、受取人の拒否権が優先される |
| **標準準拠** | W3C DID・VC・OpenID 等の国際標準に準拠し、単一主体への依存を排除する |
| **分散性** | 特定の企業・組織がプロトコルを管理・支配できない構造とする |
| **検証可能性** | 感謝の事実・価値は第三者が検証可能な形で記録される |

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| **thx_giver** | 感謝を送る人（送信者） |
| **thx_receiver** | 感謝を受け取る人（受信者） |
| **token_amount** | オンチェーンで移転するトークンの量（参照実装: HRT トークン） |
| **w_user** | オフチェーンで計算される感謝の重み（1〜100） |
| **感謝イベント** | thx_giver から thx_receiver への一つの感謝行為の記録 |
| **社会的事実（Layer 1）** | 感謝の存在・強度に関するパブリックな記録 |
| **メッセージ内容（Layer 2）** | 感謝の具体的な文章・文脈。アクセス制御対象 |
| **DID** | W3C Decentralized Identifier。ユーザーの分散型識別子 |
| **VC** | W3C Verifiable Credential。検証可能な資格情報 |

## 3. データモデル

### 3.1 感謝イベントの2層構造

感謝イベントは「社会的事実（Layer 1）」と「メッセージ内容（Layer 2）」の2層で表現される。

```
感謝イベント
├── Layer 1: 社会的事実（パブリック・改ざん不可）
│   ├── event_id          : UUID
│   ├── thx_receiver_did  : DID
│   ├── w_user            : Integer 1-100（必須）
│   │                       送信者のトークン保有量に対する移転比率から算出
│   ├── timestamp         : ISO 8601
│   ├── community_id      : String
│   └── content_hash      : SHA-256（Layer 2 への参照・改ざん検出）
│
└── Layer 2: 開示情報（アクセス制御・暗号化）
    ├── content_id        : content_hash と同値
    ├── thx_giver_did     : DID（送信者の識別子）
    │                       デフォルト: thx_receiver のみ閲覧可
    │                       送信者が秘匿を選択した場合: thx_receiver にも非公開（匿名贈与）
    ├── message_text      : String（感謝の文章）
    ├── context_text      : String（感謝の背景・文脈）
    ├── visibility        : "private" | "permitted" | "public"
    ├── permitted_dids    : [DID]（permitted の場合のアクセス許可リスト）
    └── encryption        : ECIES（thx_giver の公開鍵で暗号化）
```

### 3.2 所有権・アクセス制御ルール

| 操作 | ルール |
|------|--------|
| Layer 1 の閲覧 | 誰でも可（送信者情報を含まない） |
| thx_giver_did の閲覧（デフォルト） | thx_receiver のみ閲覧可 |
| thx_giver_did の秘匿（匿名贈与） | 送信者が選択した場合、thx_receiver にも非公開 |
| message_text・context_text のデフォルト | 非公開（thx_giver・thx_receiver のみ閲覧可） |
| Layer 2 の公開 | thx_giver・thx_receiver のどちらか一方が起点となり、相手方が同意した場合に実行 |
| Layer 2 の削除 | thx_giver・thx_receiver のどちらか一方の要求で即時実行（同意不要） |

> **非対称性の根拠**: 公開は両者の合意（どちらかが起点・どちらかが拒否できる）、
> 削除はより厳しい側が優先される。いったん公開されたコンテンツも
> 一方の要求で削除可能とすることで、忘れられる権利を保障する。

> **送信者をLayer 2に置く根拠**: 送信者と受信者の関係グラフが外部から再構築されると、
> 第三者による政治的・思想的プロファイリングに転用される危険がある（Cambridge Analytica 問題）。
> thx_giver_did を Layer 1 に含めないことで、公開データからの関係グラフ再構築を
> 構造として不可能にする。感謝の重みと受信の事実は公開するが、
> 誰から誰への感謝かという方向性はLayer 2として保護する。

### 3.3 オンチェーン記録（分散台帳）

Layer 1 のうち以下を分散台帳（ブロックチェーン）に記録する。  
具体的な内容はチェーンに乗せず、`content_hash` による参照のみとする。  
特定のブロックチェーンに依存せず、必要な性質（5.1 参照）を満たす実装を用いる。

```
オンチェーン記録:
  - token_amount の移転トランザクション
  - 送信元: サービスアカウント（委任送金機構 Section 6.3 参照）
  - 受信先: thx_receiver の識別子（DID または導出アドレス）
  - content_hash（Layer 2 への参照・改ざん検出）
  - タイムスタンプ（ブロック時刻）
```

> **ブロックチェーン上の送信者匿名性**: 個人ユーザーの識別子はオンチェーンに記録しない。
> 委任送金機構を通じてサービスアカウントが送信することで、
> ブロックチェーンのトランザクション履歴からも送信者の特定を防ぐ。

参照実装: Flow ブロックチェーン（Cadence 1.0）

### 3.4 外部システムによるデータ参照ポリシー

外部のAIシステムや第三者アプリケーションが感謝イベントデータを参照する場合、
以下のポリシーに従う。

**参照可能なフィールド（公開API）:**
- thx_receiver_did・w_user・timestamp・community_id

**参照不可:**
- thx_giver_did（Layer 2 に属し、Layer 1 に存在しない）
- 関係グラフの一括取得（送信者情報が Layer 1 に存在しないため、構造として不可能）

**w_user の必須化:**
w_user が未設定の感謝イベントはシステムが登録を拒否する。
w_user はトークン保有量に裏付けられた質の指標であり、
これを必須とすることでコストのかからないスパムイベントの大量生成を防ぐ。

**外部AIへの参照が意味するもの:**
外部システムが参照できるのは「誰が・いつ・どの程度の重みで感謝を受けたか」であり、
「誰から感謝を受けたか」ではない。これにより、信頼評価は
関係の方向性ではなく受信者の蓄積された信頼の重みとして機能する。

## 4. アイデンティティモデル

### 4.1 DID 標準への準拠

すべてのユーザーは W3C DID 標準に準拠した識別子を持つ。
本プロトコルは特定の DID メソッドを要求せず、以下の性質を満たすものを受け入れる。

| 必要な性質 | 理由 |
|-----------|------|
| W3C DID Core 1.0 準拠 | 相互運用性の確保 |
| 公開鍵の解決が可能 | 暗号署名・暗号化のため |
| DID ドキュメントが検証可能 | なりすまし防止 |
| 識別子の永続性 | 感謝の記録が長期にわたり有効であること |

準拠する DID メソッドの例（例示であり、限定ではない）：

- `did:plc` — AT Protocol ベース（参照実装での採用例）
- `did:web` — ドメインベース（組織・機関との連携に適する）
- `did:key` — 鍵ペアから生成（軽量な用途向け）
- その他 W3C 準拠の DID メソッド

### 4.2 識別子マッピング

ユーザーは複数の識別子を持ちうる。これらは DID ドキュメントを通じて紐づく。

```
DID（did:plc:xxxx）
  ├─ ブロックチェーンアドレス  ← トークン操作用（参照実装: Flow 0x...）
  ├─ Discord ID              ← コミュニティ操作用
  └─ メールアドレス等         ← 認証用（オプション）
```

## 5. トークン仕様

### 5.1 設計原則

感謝を示すトークン（例: HRT）は感謝行為をデジタルに表現するユーティリティトークンである。
特定のブロックチェーン実装に依存せず、以下の性質を満たす分散台帳上に実装できる。

| 必要な性質 | 理由 |
|-----------|------|
| 不変の取引記録 | 感謝の社会的事実として改ざん不可であること |
| スマートコントラクト対応 | 委任送金・上限制御を自律的に実行するため |
| パブリックな検証可能性 | 誰でも感謝の事実を検証できること |
| DID との統合 | ユーザーのアイデンティティとトークンを紐づけるため |

> 参照実装は Flow ブロックチェーン上の HRT トークンを採用しているが、
> 上記の性質を満たす他のブロックチェーン・分散台帳技術への移植を妨げるものではない。

### 5.3 感謝の重み計算（w_user）

感謝の「深さ」をオフチェーンで計算し、社会的信頼指標として用いる。  
オンチェーンの `token_amount` とは独立した指標である。

```
w_user の算出（1〜100のスコア）:
  - token_amount と送信者残高の比率
  - token_amount と日次配布量の比率
  → 5段階の重みレベルに変換（閾値: 0.2 / 0.4 / 0.6 / 0.8）
```

### 5.4 送金権限委任（Account Abstraction）

ユーザーはサービスアカウント（例: ragserver）に送金権限を委任できる。  
委任はスマートコントラクトレベルで制約・管理される。

```
委任制約（プロトコル要件）:
  - 1回あたりの送金上限（実装側で設定可能）
  - 日次累計上限（実装側で設定可能）
  - 委任先の検証（なりすまし防止）
  - ユーザーはいつでも即座に委任を取り消し可能
```

参照実装: HeartDelegate コントラクト（Flow Testnet `0x3e6d4975cf216943`）

```json

  "type": ["VerifiableCredential", "ContentAccessCredential"],
  "issuer": "did:plc:xxxx",
  "credentialSubject": {
    "id": "did:plc:yyyy",

  }

```

### 6.4 コミュニティメンバーシップ証明（CommunityMemberCredential）〔提案〕

コミュニティが発行する VC。この VC を提示できる人物は、そのコミュニティに属していることが証明される。

```json
{
  "@context": ["https://www.w3.org/ns/credentials/v2"],
  "type": ["VerifiableCredential", "CommunityMemberCredential"],
  "issuer": "did:web:bluemoon-community.io",
  "credentialSubject": {
    "id": "did:plc:xxxx",
    "communityId": "bluemoon",
    "role": "member",
    "joinedAt": "2026-01-15",
    "validUntil": "2026-12-31"
  }
}
```

この VC は Section 7.1 のアクセス制御と組み合わせることで、
**コミュニティ単位の Layer 2 可視範囲制御**を実現する（Section 7.3 参照）。

## 7. RAG（検索拡張生成）統合

### 7.1 プライバシーフィルタリング

感謝イベントの Layer 2 コンテンツを RAG データとして利用する際は、
`visibility` フィールドに基づいてフィルタリングを行う。

```
クエリ時のフィルタリング:
  public      → 誰でも検索可
  permitted   → requester_did が permitted_dids に含まれる場合のみ
  vc_gated    → requester が required_vc_type の VC を提示できる場合のみ（提案）
  private     → owner（thx_giver / thx_receiver）本人のみ
```

### 7.2 ベクトルデータとコンテンツの分離（中長期）

```
Qdrant（ベクトルDB）:
  - 埋め込みベクトル
  - メタデータ: { content_id, visibility, permitted_dids, owner_did, community_id }
  - ※ 原文テキストは保存しない（content_id で参照）

暗号化コンテンツストア:
  - content_id をキーに暗号化済み原文を保存
  - thx_giver の公開鍵（DID に紐づく）で ECIES 暗号化
```

### 7.3 VC ベースのコミュニティアクセス制御〔提案〕

`visibility: "vc_gated"` を導入し、特定の VC を保持するユーザーグループへのアクセスを制御する。

感謝イベント（Layer 2）のメタデータ拡張:

```json
{
  "content_id": "sha256:abc...",
  "visibility": "vc_gated",
  "vc_gate": {
    "required_vc_type": "CommunityMemberCredential",
    "required_vc_issuer": "did:web:bluemoon-community.io",
    "required_community_id": "bluemoon"
  }
}
```

**アクセスフロー:**

```
1. ユーザーが Layer 2 コンテンツをリクエスト
2. RAG サーバーが visibility: "vc_gated" を確認
3. ユーザーに CommunityMemberCredential の提示を要求
4. RAG サーバーが VC の署名・発行者・有効期限を検証
5. communityId が一致すればアクセス許可
```

**ユースケース例:**

- コミュニティ内の感謝メッセージを、そのコミュニティメンバーだけが互いに閲覧できる
- 特定の役割（contributor, moderator 等）を持つメンバーにのみ詳細コンテキストを開示する
- コミュニティをまたいだ連携（複数 VC の AND/OR 条件）

**実装上の注意:**

- VC の検証は Section 8.2（VC 検証エンドポイント）を用いる
- Qdrant メタデータへの `vc_gate` フィールド追加が必要（Phase A 対応範囲）
- VC 検証ロジックは Phase C として実装予定

## 8. AIエージェントと RAG の統合

感謝経済プロトコルは、AIエージェントを自然言語インターフェースとして活用することを
標準的な実装パターンとして想定する。

AIエージェントは RAG サーバー（感謝イベントのベクトルDB）と連携し、
コミュニティ内の関係性・贈与の文脈を参照しながら、
ユーザーとの対話を通じて感謝を記録・伝達する。

```
ユーザー（音声・テキスト）
    ↓
AIエージェント ─── RAG サーバー（感謝イベント・コンテキスト）
    ↓
感謝イベントの構造化・記録（Layer 1 / Layer 2）
    ↓
受取人への通知・伝達
```

AIエージェントの形態は問わない（テキストボット・音声アシスタント・
物理キオスク・VR アバター等）。いずれも同一のプロトコルに従う。

## 9. 外部システム連携インターフェース

外部システムは以下の標準プロトコルを通じて接続できる。  
本プロトコルの内部実装を変更することなく、標準準拠システムは相互運用可能である。

### 8.1 DID 解決

```
GET /.well-known/did.json  （did:web の場合）
または AT Protocol PDS     （did:plc の場合）
→ DID ドキュメントを返す（公開鍵・サービスエンドポイント）
```

### 8.2 VC 発行・検証

```
POST /credentials/issue   → VC の発行（JWT / SD-JWT 形式）
POST /credentials/verify  → VC の検証
POST /credentials/present → VP（Verifiable Presentation）の提示
```

標準: W3C VC Data Model 2.0 / OpenID for Verifiable Credentials（OID4VC）

### 8.3 感謝イベント API

```
POST /events/thanks        → 感謝イベントの記録
GET  /events/{event_id}    → Layer 1 データの取得（パブリック）
GET  /events/{event_id}/content → Layer 2 取得（アクセス制御あり）
GET  /users/{did}/stats    → ユーザーの社会的事実サマリー
```

## 10. プライバシー原則まとめ

1. 感謝の**社会的事実**（存在・量・強度）は原則パブリックとする
2. 感謝の**メッセージ内容**は送信者が所有し、受取人の拒否権が優先される
3. ユーザーは自身のデータに対して**閲覧・変更・削除・移転**の権利を持つ
4. サービス事業者はユーザーの暗号化済みコンテンツを**復号できない**設計を目指す
5. コンテンツへのアクセス許可付与・取り消しは**本人がいつでも実行可能**とする

## 11. オープンガバナンス原則

1. 本プロトコル仕様は特定の企業・組織に帰属しない
2. 仕様の変更提案はオープンな場（GitHub Issues 等）で議論する

## 12. データ透明性と自律性の保障

### 12.1 基本原則

RAGを用いた信頼評価システムは、適切な設計がなされなければ監視・格付け・排除の
自動化につながる危険性がある。本プロトコルに準拠するシステムは、
この危険に対する構造的な防衛として以下の権利と義務を実装しなければならない。

### 12.2 ユーザーの権利

| 権利 | 内容 |
|------|------|
| **自己データ照会権** | 自身の DID に紐づく全データの一覧と、AIによる自然言語解釈を**要求から30日以内**に受け取ることができる |
| **ベクトルデータ可読化の要求権** | ベクトル化されたデータが何を示しているかを、AI を用いて人間が理解できる形で説明させることができる |
| **信頼評価生成の説明要求権** | 自身に対する評価・スコア・推薦結果について、**上位3つの根拠要因**を明示した説明を受け取ることができる |
| **訂正・削除権** | 誤りを含むデータの訂正、および Layer 2 データの削除を**要求から72時間以内**に実行させることができる（Section 3.2 参照） |
| **データポータビリティ権** | 自身のデータを機械可読な形式（JSON-LD 等）で取得・移転できる |

> 上記の権利は、EU 一般データ保護規則（GDPR）第15〜20条、および
> 日本個人情報保護法に基づく権利と整合する[^2]。

### 12.3 システムの義務

1. **自己データ照会エンドポイントの提供** — 以下の仕様を満たすエンドポイントを実装しなければならない
   - `GET /users/{did}/data-report` — DID に紐づく全データ一覧と AI 解釈を返す
   - 応答に「このデータがどのような評価に使われているか」の説明を含む
   - 認証された本人以外はアクセス不可

2. **自動意思決定の説明義務** — 信頼評価・スコア・推薦等の自動処理結果に対して、
   **上位3つの根拠要因を自然言語で**提示しなければならない

3. **禁止される利用（具体的列挙）** — 以下の目的への利用を明示的に禁止する
   - 雇用・融資・保険等の審査への自動的な組み込み
   - 物理空間へのアクセス制限・排除への利用
   - 政治的・宗教的・思想的傾向の推論・記録
   - ユーザーの同意なき第三者提供
   - ユーザーの行動を変容させることを目的とした介入（ナッジング）

4. **年次透明性レポートの公開義務** — 毎年1回以上、以下を公開しなければならない
   - 処理した感謝イベント総数（個人特定不可の形で）
   - 自己データ照会要求の件数と対応率
   - 削除要求の件数と実行率
   - 説明要求の件数と対応率

### 12.4 信頼評価システムへの準拠要件

本プロトコルを用いて信頼評価を提供するシステムが「準拠」を名乗るためには、
上記 12.2・12.3 の全項目を実装していることが条件となる。

これは、信頼評価の提供を「便利な機能」としてではなく、
**ユーザーの自律性を前提とした権利関係**として設計することを義務づけるものである。
